设计模式-装饰模式

发表于 2021-01-15 分类于 设计模式

装饰模式decorator

by andyau

  • 什么是装饰模式/装饰器模式？

  • *动态（组合）地给一个对象增加一些额外的职责**。就增加功能而言，装饰模式比生成子类（使用继承）更为灵活（消除重复代码以及减少子类个数）。——《设计模式》GoF

  • 单一职责
    在软件组件的设计中，**如果责任划分不明确，使得继承得到的结果往往是随着需求的变化而导致子类急剧膨胀，同时充斥着重复代码，这时候的关键是划清各部分的职责**。
    使用“单一职责”模式的设计模式，典型的有Decorator和Bridge.

  • 要点

    • 要点一
      通过采用组合而非继承的手法，装饰模式实现了在运行时动态扩展对象功能的能力，而且可以根据需要扩展多个功能。避免了使用继承带来的“灵活性差”和“多子类衍生问题”。
    • 要点二

    Decorator类在接口上表现为is-a Component的继承关系，即Decorator类继承了Component类的所具有的接口。但在实现上又表现为has-a Component的组合关系，即Decorator类又使用了另外一个Component类。

    • 要点三
      Decorator模式的目的并非解决“多子类衍生的多继承”问题。而是在于解决“主体类在多个方向上的扩展功能”——这就是“装饰”的意思。

    • 比如有一个手机，允许你为手机添加特性，比如增加挂件、屏幕贴膜等。

设计模式-观察者模式

发表于 2021-01-15 分类于 设计模式

观察者模式Observer

by andyau

  • 什么是观察者模式？
    又叫“Event”。在软件构建过程中，我们需要为某些对象建立一种通知依赖关系——一个对象（目标对象）的状态发生改变时，所有的依赖对象（观察者对象）都将得到通知。如果这样的依赖关系过于紧密，将使软件不能很好地抵御变化。
    使用OOP技术，可以将这种依赖关系弱化，并形成一种稳定的依赖关系，从而实现软件体系结构的松耦合。
    通俗来讲，即定义对象间的一种一对多的依赖关系，以便当一个对象的状态发生改变时，所有依赖于它的对象都得到通知并自动更新。——《设计模式》GoF

  • 观察者模式四大角色

    • 抽象主题角色(Subject)
      抽象主题角色提供维护一个观察者对象集合的操作方法，对集合的增加、删除等。
    • 具体主题角色(ConcreteSubject)
      将有关状态存入具体的观察者对象；在具体主题的内部状态改变时，给所有登记过的观察者发通知。具体主题角色负责实现抽象主题中的方法。
    • 抽象观察者角色(Obeserver)
      为具体观察者提供一个更新接口。
    • 具体观察者角色(ConcreteObserver)
      存储与主题有关的自洽状态，实现抽象观察者提供的更新接口。

  • 例子

    • 在教室里老师还没有来，同学都在干着各的事情，小张正在打游戏，小李正在抄作业….., 现在同学们要求班长当卧底，监视老师，当老师来了通知大家一声。然后打游戏的马上停止，抄作业的也停止。这里班长相当于是一个通知者， 小张、小李，以及其他同学显然是监听者，他们监听了班长那的消息，一旦老师来了马上采取相关的行动。

  • 要点

    • 要点一
      使用面向对象的抽象,观察者模式使得我们可以独立地改变目标与观察者，从而使二者之间的依赖关系松耦合。
    • 要点二
      目标发送通知时，无需指定观察者，通知（可以携带通知信息作为参数）会自动传播。
    • 要点三
      观察者自己决定是否需要订阅通知，目标对象对此一无所知。

策略模式Strategy

by andyau

  • 什么是策略模式？

    定义一系列算法，把他们一个个封装起来，并且是他们可互相替换；该模式使得算法可独立于使用它的客户程序（稳定）而变化（扩展、子类化）。——《设计模式》GoF

  • 要点

    • 要点一
      Strategy及其子类为组件提供了一系列可重用的算法，从而使得类型在运行时方便地根据需要在各个算法之间进行切换。
    • 要点二
      Strategy模式提供了用条件判断语句以外的另一种选择，消除条件判断语句，就是在解耦合。很有许多条件判断语句的代码通常都需要用策略模式。
    • 要点三
      如果Strategy对象没有实例变量，那么各个上下文可以共享一个Strategy对象，从而节省对象开销。

  • 例子

    • 有一个“出行旅游”的基策略类TravelStrategy.cpp，其中有一个“出行”方法vehicle()；
    • 有三个具体策略类：飞机出行、火车出行、单车出行，都继承了基策略类，并实现了“出行”方法。
    • 有一个环境类（在此例中是用户类PersonContext），作为解决类提供用户选择出行解决方案.
    • 在Java里，基策略类可以是抽象类或者是接口类。
